\begin{abstract}
We present \textit{Explainable Chronos}, a modular NLP framework built on top of Chronos-2 that makes probabilistic time-series forecasts interpretable and interactively queryable. Extension~1 generates natural-language summaries of quantile forecasts by using  extracted numerical features and covariate attributions via Attention Rollout, and evaluates each sentence for factual consistency using NLI. Extension~2 adds a conversational interface through which users can query and modify forecasts in plain English usinh a three-tier intent parser (rule-based, SBERT few-shot, LLM) classifies queries into five intent types, applies input modifications, re-runs Chronos-2, and returns a new verbalized response. Evaluated on 200 windows of the Weather dataset, Extension~1 achieves mean NLI consistency of 0.7333 (analyst persona) and 0.6858 (executive persona), with semantic similarity to the template reference of 0.8640 and 0.7893 respectively; On a 60-query test set, Extension~2 achieves 93.3\% intent accuracy, 97.2\% covariate resolution, and 83.3\% task completion rate; the SBERT tier (Tier~2) outperforms the rule-based tier on its subset (94.7\% vs.\ 92.5\%), and multi-turn state persistence holds at 100\%.
\end{abstract}

\begin{IEEEkeywords}
time-series forecasting, natural language generation, explainability, NLI consistency, dialogue systems, intent parsing, Chronos-2, covariate attribution

\section{Introduction}
% =====================================================================

Chronos-2~\cite{chronos2} is a probabilistic time-series forecasting model that outputs quantile predictions capturing trends, uncertainty, and tail risks across a forecast horizon. Beyond the raw forecasts, the transformer attention layers Chronos-2 uses to process historical data and covariates create a natural starting point for explainability: they record, implicitly, which parts of the input history and which covariates drove the prediction. The problem is that most users, especially non-technical ones, cannot exploit any of that. A set of quantile curves gives no intuition about why the forecast looks the way it does, how reliable it is, or which input variable matters most. Our work builds a NLP layer on top of Chronos-2 to address this. Extension~1 translates the model's quantile output into plain-language explanations grounded in actual numerical features, so non-expert users can understand what the forecast is saying and why. Extension~2 adds a conversational interface where users can ask questions and request modifications in plain English, and the system re-runs Chronos-2 and explains the updated result.

\section{Problem Statement}
% =====================================================================

\subsection{Extension 1}

\textbf{Input.}
A historical time series $\mathbf{x} = (x_1, \dots, x_T) \in \mathbb{R}^T$ with a matrix of covariates $\mathbf{Z} \in \mathbb{R}^{T \times C}$ with $C$ named covariates.

\vspace{2mm}

\textbf{Task.}
Given a Chronos-2 quantile forecast $\hat{\mathbf{y}}^{(q)}$ for quantiles $q \in \{0.10, 0.50, 0.90\}$ over a horizon $H$:
%
\begin{equation}
  \hat{\mathbf{y}}^{(q)} = \bigl(\hat{y}^{(q)}_1, \dots, \hat{y}^{(q)}_H\bigr), \quad q \in \{0.10, 0.50, 0.90\},
\end{equation}
%
produce a natural-language summary $\mathcal{S} = (s_1, \dots, s_K)$ such that each sentence $s_k$ is factually grounded in a subset of numerical features derived from the forecast.

\vspace{2mm}

\textbf{Output.}
A natural-language verbalization $\mathcal{S} = (s_1, \dots, s_K)$ of the quantile forecast.

\subsection{Extension 2}

\textbf{Input.}
A natural-language query $q$ issued by a user, together with the current dialogue state comprising a covariate set $\mathbf{Z} \in \mathbb{R}^{T \times C}$ and a forecast horizon $H$.

\vspace{2mm}

\textbf{Task.}
Classify $q$ into one of five intent types $\mathcal{I} = \{$\texttt{remove\_covariate}, \texttt{scale\_covariate}, \texttt{change\_horizon}, \texttt{confidence\_query}, \texttt{counterfactual}$\}$, extract any relevant slots (target covariate, scale factor, new horizon), apply the corresponding modification to the forecast inputs, re-run Chronos-2, and generate a natural-language response via the Extension~1 verbalization pipeline. State is preserved across turns.

\vspace{2mm}

\textbf{Output.}
A dialogue response comprising: (a) a human-readable description of the modification applied; (b) a new natural-language forecast summary; (c) the delta P50 (change in median forecast relative to the unmodified baseline).

\section{Methodology}
% =====================================================================

\subsection{Extension 1}

The pipeline consists of three sequential stages: forecast and feature extraction (1), covariate attribution (2), and verbalization (3).

\subsubsection{Forecast and Feature Extraction}

From the Chronos-2 quantile output (P10, P50, P90) we extract a set of interpretable numerical features (covering trend direction and magnitude, uncertainty level and evolution, tail risks, regime shifts, and trajectory shape) that feed directly into the verbalization step.




% ----- 4.2 -----
\subsubsection{Covariate Attribution}

The pipeline computes covariate attribution scores via Attention Rollout~\cite{abnar2020attentionflow} and passes the top-$k$ results (default $k=5$) to the verbalizer.

\textbf{Method.}
~\cite{abnar2020attentionflow} observes that raw per-layer attention weights become near-uniform in deep transformers, as each layer's output is a mixture of all previous representations.
Attention Rollout addresses this by composing attention matrices across layers: at each layer the raw weights are averaged over heads, mixed with an identity term to account for residual connections ($A = 0.5 \cdot W_{\text{att}} + 0.5 \cdot I$), and left-multiplied into the accumulated product ($\tilde{A}(l_i) = A(l_i) \cdot \tilde{A}(l_{i-1})$).
The final matrix $\tilde{A}$ approximates how information from each input position propagates to the output.

\textbf{Adaptation to Chronos-2.}
Chronos-2 produces two attention tensors that we exploit for different questions.
\texttt{enc\_group\_self\_attn\_weights} (shape $\text{Patches} \times \text{Heads} \times \text{Series} \times \text{Series}$) tells us how each series attends to every other series, we use this to answer \emph{which covariate matters most}.
\texttt{enc\_time\_self\_attn\_weights} (shape $\text{Series} \times \text{Heads} \times \text{Patches} \times \text{Patches}$) tells us how each series attends to its own history, we use this to answer \emph{which past time steps each covariate focused on}.
Throughout, series index~0 is the target and indices $1, \ldots, C$ are the covariates.

\emph{Covariate importance.}
We want a single number per covariate: how much did it influence the target?
We start by collapsing the group-attention tensor at layer $l$ into one matrix by averaging over patches and heads:
%
\begin{equation}
  \bar{W}^{(l)} = \frac{1}{PH}\sum_{p,h} W^{(l)}_{p,h,:,:} \;\in\; \mathbb{R}^{S \times S}.
\end{equation}
%
Attention Rollout is then applied across all $L$ layers ($A^{(l)} = 0.5\,\bar{W}^{(l)} + 0.5\,I$, $\tilde{A}^{(l)} = A^{(l)} \cdot \tilde{A}^{(l-1)}$), and the answer we need sits in row~0 of the final matrix: $\tilde{A}^{(L)}_{0,c}$ is how much covariate $c$ contributed to the target.
We normalise these scores so they sum to 100\%:
%
\begin{equation}
  \hat{\alpha}_c = \frac{\tilde{A}^{(L)}_{0,c}}{\sum_{c'=1}^{C}\tilde{A}^{(L)}_{0,c'}}.
\end{equation}

\emph{Temporal saliency.}
For covariate importance we looked \emph{across} series; for temporal saliency we look \emph{within} each series: which part of its own past did covariate $c$ attend to most? We apply rollout to the time-attention layers for each covariate separately (slice $c$ of the Series dimension; slice~0 is the target), obtaining a $(P \times P)$ rollout matrix $\tilde{T}^{(L)}_c$.
Each column $p$ of this matrix corresponds to a history patch, and each row $q$ to a current patch; entry $[q,p]$ encodes how much history patch $p$ contributed to current patch $q$.
We do not care \emph{which} current patch was affected rather we want the total importance of each history patch, so we average across all rows:
%
\begin{equation}
  \sigma_c[p] = \frac{1}{P}\sum_{q}\tilde{T}^{(L)}_c[q,\,p].
\end{equation}
%
$\sigma_c[p]$ is now a single score per history patch: how much did the model rely on patch $p$?
Because each patch covers multiple consecutive time steps, this score lives at patch granularity, not at the original time-step level.
We therefore re-sample $\sigma_c$ to the original $T$ time steps by spreading each patch score across the time steps it covers, then normalise the result to sum to one.
From the resulting distribution we extract the \emph{peak step} $t^{*} = \arg\max_t\,\sigma_c[t]$ and the \emph{focus breadth}
%
\begin{equation}
  b_c = \frac{-\sum_{t}\sigma_c[t]\log\sigma_c[t]}{\log T} \;\in\; [0,\,1],
\end{equation}
%
the normalised entropy: low means attention was tight around a short window, high means it was spread evenly.

% ----- 4.3 -----
\subsubsection{Verbalization}

We implement two complementary verbalizers.

\textbf{Template Verbalizer.}
A deterministic module that maps each active feature group to a fixed sentence, producing one sentence per feature (trend, trajectory, uncertainty, conditional risks). A rhetorical layer then adds RST-structured connectives~\cite{mann1988rst} (concession, contrast, elaboration, cause) when feature combinations call for them.

\textbf{LLM Verbalizer.}
The template output acts as a structured draft that a LLM rewrites via few-shot in-context learning. The prompt supplies the numerical features, grounding triples, the draft, and explicit constraints against causal speculation and numerical deviation. Greedy decoding keeps the output deterministic. Two audience modes are supported: \emph{analyst} (uses more technical terminology) and \emph{executive} (plain language for non-technical readers), each with a distinct system prompt and few-shot set while sharing the same model weights. 

\subsection{Extension 2}

Extension~2 wraps Extension~1 in a stateful conversational interface, letting users query and modify the forecast in plain English across multiple turns.

\subsubsection{Intent Parser}

Queries are classified using a three-tier pipeline.

\textbf{Tier 1: rule-based parser.}
A catalog of regex patterns with explicit priority ordering. When a pattern fires, slot extraction resolves the relevant details: the target covariate via exact match, token overlap, or substring fallback; the scale factor via keywords such as \textit{double} or \textit{halve}, or a percentage; and the new horizon via digit or word-number patterns.

\textbf{Tier 2: SBERT few-shot classifier.}
When no rule matches, a sentence-BERT model~\cite{reimers2019sbert} is used to classify the query by nearest-neighbour search a pool of 100 labeled examples. The query and every example are embedded and compared by cosine similarity; the top-$k$ neighbours vote on the intent type. If the best similarity falls below a confidence threshold, the query is passed to Tier 3. Slot extraction then runs on the predicted intent as in Tier 1. 

\textbf{Tier 3: LLM fallback.}
If Tier 2 is also uncertain, a LLM produces a structured JSON intent classification as a last resort. 

\subsubsection{Input Modifier}

Once an intent is classified, the system applies it to the current forecast inputs, producing a modified copy without altering any persistent state. A \texttt{remove\_covariate} intent zeros out the target covariate channel; \texttt{scale\_covariate} multiplies its values by the extracted factor; \texttt{change\_horizon} updates the forecast window length. Covariate names are matched flexibly via token overlap to handle aliases and partial names.


\subsubsection{Response Generation}

For modifying intents, the system re-runs the full Extension~1 pipeline on the updated inputs, including Attention Rollout attribution, and returns a new plain-language summary together with the delta P50 relative to the unmodified baseline.

For \texttt{confidence\_query}, the system answers directly from the current forecast's P10/P50/P90 without re-running the model, since no inputs changed.

For \texttt{counterfactual}, the system declines the modification. Because the forecasting model treats the historical window as fixed ground truth to establish the current state, modifying past inputs would require a complex causal simulation of an alternate timeline up to the present. Instead, the system redirects the user toward a forward-looking alternative.

\section{Experiments}
% =====================================================================

\subsection{Extension 1}

% ----- 6.1 -----
\subsubsection{Data}

We evaluate on the Seattle Weather dataset~\cite{vega_seattle_weather}.
The forecast target is daily maximum temperature (\texttt{temp\_max}), with three meteorological covariates: minimum temperature (\texttt{temp\_min}), precipitation (\texttt{precipitation}), and wind speed (\texttt{wind}).

% ----- 6.2 -----
\subsubsection{Experimental Design}

\textbf{Hardware.} NVIDIA A100 GPU via Google Colab.

\textbf{Software and libraries.} Python~3.10+; \texttt{chronos-forecasting} (\texttt{Chronos-2-small}~\cite{chronos2}); \texttt{transformers} (\texttt{BART-large-MNLI}~\cite{lewis2020bart} for NLI, \texttt{Qwen2.5-7B-Instruct}~\cite{qwen} for LLM verbalization); \texttt{sentence-transformers} (\texttt{all-MiniLM-L6-v2} for semantic similarity scoring).

\textbf{Validation method.}
We evaluate over 200 scenarios on the test split, each using a history window of 512 steps and a forecast horizon of 96 steps, positioned at evenly spaced starting points. Both LLM personas (\emph{analyst}, \emph{executive}) share the same weights and differ only in system prompt and few-shot examples. Forecast accuracy metrics are computed against observed actuals; verbalization quality metrics are computed against the model's own extracted numerical features.

\textbf{Performance metrics.}
Two orthogonal questions are evaluated: (1)~\emph{Is the forecast accurate?} and (2)~\emph{Is the verbalization a faithful and complete description of what the model predicted?}
Forecast accuracy (against observed actuals, independently of the verbalization pipeline): \textbf{MASE} (ratio of P50 MAE to a na\"ive one-step baseline), \textbf{Fair MASE} (ratio of P50 MAE to a flat multi-step naive baseline), and \textbf{Coverage} (fraction of actuals inside the P10--P90 interval).
Verbalization quality (LLM verbalizer only): \textbf{NLI consistency} (per-sentence entailment probability from BART-large-MNLI, averaged over all $K$ sentences) measures faithfulness; \textbf{semantic similarity vs.\ template} (cosine similarity between SBERT~\cite{reimers2019sbert} embeddings of the LLM output and the template draft) measures semantic drift.

% ----- 6.3 -----
\subsubsection{Results}

\paragraph*{a) Forecast Accuracy}
Over 200 evaluation windows, \texttt{Chronos-2-small} achieves a mean MASE of $0.998 \pm 0.184$ and a Fair MASE of $0.425 \pm 0.167$, indicating that the median forecast substantially outperforms a flat na\"ive baseline over the 96-step horizon. The P10--P90 interval achieves $91.4\% \pm 4.6\%$ coverage.

\paragraph*{b) Verbalization Quality}

Table~\ref{tab:verbalization} reports NLI consistency and semantic similarity for both personas across 200 Weather windows. The analyst persona uses a lexicon with less paraphrasing that stays closer to the underlying NLI premises, which helps it maintain a relatively high mean NLI consistency of 0.7333. In contrast, the executive persona scores lower on NLI consistency (0.6858).

\subsection{Extension 2}

\subsubsection{Data}

The multi-turn evaluation uses a 100-step window from the ETTh1 dataset~\cite{zhou2021informer} with ten named covariates. For intent-only evaluation no time series data is needed.

\subsubsection{Experimental Design}

\textbf{Hardware.} NVIDIA A100 GPU via Google Colab.

\textbf{Software and libraries.} Python~3.10+; \texttt{chronos-forecasting} (\texttt{Chronos-2-small}) for re-running the forecast after input modifications; \texttt{sentence-transformers} (\texttt{all-MiniLM-L6-v2}) for Tier~2 SBERT few-shot classification; \texttt{transformers} (\texttt{Qwen1.5-1.8B-Chat}) as the Tier~3 LLM fallback; Python \texttt{re} for the Tier~1 rule-based patterns.

\textbf{Validation method.}
A \textbf{development set} (100 queries) was used to build the regex patterns and serves as the labeled neighbour pool for SBERT few-shot inference. The \textbf{test set} (60 queries, written after all patterns and the development set were frozen) covers three difficulty levels: 17 standard phrasings, 23 ambiguous cases, and 20 outlier phrasings designed to test Tier~2 and Tier~3. Multi-turn state persistence is verified separately by running three two-turn sequences and checking that modifications from earlier turns remain in effect.

\textbf{Performance metrics.}
Evaluation focuses on the intent parsing pipeline: \textbf{Intent accuracy} (correct intent type), \textbf{covariate resolution accuracy} (correct covariate extracted, for queries that require one), and \textbf{Task Completion Rate} (TCR, conjunction of both) Multi-turn state persistence is verified as a binary check per sequence.

\subsubsection{Results}

Table~\ref{tab:ext2_metrics} reports overall performance on the 60-query test set and per-intent accuracy. The system achieves 93.3\% intent accuracy and 97.2\% covariate resolution (35/36 queries with a covariate slot), for a Task Completion Rate of 83.3\%. Three intent types reach 100\% accuracy. The two weaker types are \texttt{remove\_covariate} (78.6\%, 3 failures) and \texttt{counterfactual} (83.3\%, 1 failure); all failures are analysed in Appendix~B.

\textbf{Extension 1: Illustrative Success and Failure Cases}

\textbf{Success case}
The extracted features indicate a flat trend ($+0.0415$ per step), high and widening uncertainty, both downside and upside risk, and a statistically significant regime shift ($p < 10^{-5}$). The dominant covariate is \texttt{temp\_min} at 39.0\%, followed by \texttt{precipitation} (30.8\%) and \texttt{wind} (30.2\%).
The LLM produces:
\begin{quote}
\emph{``The forecast suggests stability over the next 96 periods, starting from 22.20 and ending near 27.34, which is 23.2\% higher. Prediction intervals are wide and expanding, indicating increasing uncertainty. There is both downside and upside risk, with bounds up to 20\% above or below current levels. A structural break is detected mid-horizon, driven mainly by temperature (39.0\%), followed by precipitation (30.8\%) and wind (30.2\%).''}
\end{quote}
Every numerical claim and directional fact maps directly to a feature in the grounding, and the summary is factually consistent.

\textbf{Failure case}
Suppose the LLM had instead produced:
\begin{quote}
\emph{``The forecast indicates a declining trend over the 96-period horizon, likely driven by adverse seasonal weather patterns and temperature inversions. Long-term conditions appear to be deteriorating, suggesting the structural shift observed at the midpoint may persist well beyond the forecast window.''}
\end{quote}
The NLI premise for the first sentence contains only the flat trend classification and a slope of $+0.0415$; no mention of ``declining,'' ``seasonal patterns,'' or ``temperature inversions.''
The NLI model assigns an entailment probability near~0.05, correctly flagging the direction reversal (flat vs.\ declining) and the hallucinated domain-specific causal content.
This illustrates the key failure mode of the LLM verbalizer: despite the grounding constraints in the prompt, the model can introduce plausible-sounding but factually unsupported language that the NLI metric reliably catches.

\textbf{Appendix B: Extension 2 — Failure Analysis}

The evaluation on the 60-query test set produced 4 failures. They fall into two root causes: rule priority conflicts and vocabulary gaps.

\textbf{Failure 1 — Rule priority conflict (scale fires before counterfactual).}
Query: \textit{``What if sales yesterday had been 20\% higher?''}
Expected: \texttt{counterfactual}. Predicted: \texttt{scale\_covariate}.
The phrase ``20\% higher'' matches the scale pattern \texttt{r"\textbackslash b\textbackslash d+\textbackslash s*\%\textbackslash s+higher\textbackslash b"} before the counterfactual check is reached. Because \texttt{scale\_covariate} sits above \texttt{counterfactual} in the priority ordering, the temporal marker ``yesterday'' is never examined. Fix: add a past-tense guard to scale patterns, or lower the priority of percentage patterns when strong past-tense markers are present.

\textbf{Failure 2 — Vocabulary gap (remove verb not in catalog).}
Query: \textit{``What if we paused all advertising?''}
Expected: \texttt{remove\_covariate}. Predicted: \texttt{counterfactual}.
The verb ``paused'' is not in the remove-covariate pattern catalog. No remove or scale pattern fires, and the query falls through to counterfactual via the pattern \texttt{r"\textbackslash bwhat\textbackslash s+if\textbackslash b"}. Fix: add ``pause'', ``suspend'', ``disable'' and similar verbs to the remove catalog.

\textbf{Failure 3 — BERT misclassification (novel verb outside embedding neighbourhood).}
Query: \textit{``Kill the shipping delays.''}
Expected: \texttt{remove\_covariate}. Predicted: \texttt{scale\_covariate}.
``Kill'' is not in any rule pattern, so the query reaches Tier~2. Despite its semantic similarity to removing/zeroing a variable, the BERT nearest-neighbour search associates it with scaling examples (likely because ``kill'' and ``cut'' share embedding space with scale-down phrasings). Fix: add ``kill'', ``eliminate'', ``wipe'' to the remove catalog in Tier~1 so the query never reaches BERT.

\textbf{Failure 4 — Vocabulary gap (counterfactual pattern too broad).}
Query: \textit{``What happens if we ditch the weather input?''}
Expected: \texttt{remove\_covariate}. Predicted: \texttt{counterfactual}.
``ditch'' is not a remove keyword; the query falls through to counterfactual via the pattern \texttt{r"\textbackslash bif\textbackslash s+we\textbackslash b"}. Fix: add ``ditch'', ``scrap'', ``ignore'' to the remove catalog, or tighten the counterfactual pattern to require a past-tense auxiliary (``had'', ``would have'', ``were'').
